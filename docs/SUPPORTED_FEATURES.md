# Supported Features

## Loader

- ZIP packages.
- Browser folder input.
- `.def` discovery in root or nested character folder.
- Case-insensitive virtual path resolution.
- UTF-8/BOM text decoding with Windows-1252 fallback.

## DEF

- `[Info]`: `name`, `displayname`, `versiondate`, `mugenversion`, `author`.
- `[Files]`: `cmd`, `cns`, `st`, `stcommon`, `sprite`, `anim`, `sound`, `pal1` through `pal12`.
- Raw sections and diagnostics.

## AIR

- `[Begin Action N]`.
- Frames: `group, image, x, y, time`, optional `flip`, optional `blend`.
- `Loopstart`.
- `Clsn1`, `Clsn2`, `Clsn1Default`, `Clsn2Default`.

## SFF

- Header/version detection.
- Initial SFF v1 subfile parsing.
- Initial 8-bit, single-plane PCX RLE decoding.
- SFF v1 palette propagation for normal and previous-palette sprites.
- SFF v1 `samePalette` PCX subfiles without embedded palette blocks decode against the previous palette instead of truncating the RLE stream.
- Linked SFF v1 sprite records.
- Decoded SFF v1 sprites are exposed through `SffSpriteProvider` and can render in Inspector Mode.
- SFF v2 header, sprite table, palette table, format counts, and palette records.
- SFF v2 RAW8/RAW24/RAW32 decoding.
- SFF v2 RLE8 indexed-sprite decoding.
- SFF v2 RLE5 and LZ5 5-bit indexed-sprite decoding.
- SFF v2 unsupported format counts are surfaced in diagnostics and compatibility reports.
- Unsupported or undecodable sprites fall back to mock sprites through `SpriteProvider` so inspection can continue.

## SND

- Header/version detection for `ElecbyteSnd\0` archives.
- Linked subfile table parsing with group/index lookup.
- Embedded RIFF/WAVE payload extraction.
- Basic WAV metadata: channels, sample rate, and bits per sample.
- Parsed SND archives are exposed in Inspector/Compatibility UI.
- `AudioEventSystem` converts partial `PlaySnd`/`StopSnd` controllers into bounded `RuntimeSoundEvent` snapshot history with parsed `Sgroup,index`, channel, state, and tick telemetry.
- Runtime `PlaySnd` events can decode and play matching WAV payloads through Web Audio after browser user-gesture unlock.
- `StopSnd` can stop tracked channel playback in the partial Web Audio path.

## Stage DEF

- Stage `.def` discovery from ZIP/folder packages when files live under `stages/`.
- `[Info]`, `[Camera]`, `[PlayerInfo]`, `[StageInfo]`, `[Music]`, `[BGDef]`, and `[BG ...]` section parsing.
- Imported stage runtime mapping for display name, side bounds, camera start/zoom, player starts/facing, zoffset, and localcoord.
- Stage sprite/music path resolution from the stage folder, package root, and `data/` style fallbacks.
- Parsed `[BG ...]` sections preserve `id`, `spriteno`, `actionno`, `start`, `delta`, `layerno`, `tile`, and `tilespacing` metadata.
- `[BGCtrlDef ...]` and `[BGCtrl ...]` sections are parsed into Stage BG controller IR with group name, inherited `looptime`, inherited/per-controller `ctrlID`, controller type, timing window, params, and target layer ids.
- Recognized BGCtrl types are `Null`, `Visible`, `Enabled`, `VelSet`, `VelAdd`, `PosSet`, `PosAdd`, `Anim`, `SinX`, and `SinY`. They execute through a bounded renderer path for visibility/enabled skips, position/velocity offsets, action swaps, and simple sine offsets. Exact MUGEN timing, side effects, window/mask/trans behavior, and full parity remain unsupported.
- Static normal BG sprites can render from the stage SFF when the referenced `spriteno` is decoded by the existing SFF reader.
- Embedded stage `[Begin Action N]` blocks are parsed for action-backed BG layers.
- Action-backed BG layers can render when their referenced AIR frames resolve to decoded stage SFF sprites.
- Basic horizontal camera parallax is applied from layer `delta`.
- Basic bounded sprite tiling is applied from `tile`/`tilespacing` so repeated floor/wall layers can fill the viewport.
- Stage compatibility summaries report DEF/SFF/music presence, decoded stage sprites, BG sprite coverage, animated BG coverage, tiled layers, bounded BGCtrl coverage, placeholder fallbacks, warnings, and known unsupported stage features.
- Stage compatibility reports now include a first per-layer BG IR: each imported `[BG ...]` layer records its source section, type, layer number, control id, start, delta, tiling, sprite/action reference, decoded-frame coverage, unsupported layer features, fallback reason, and status as `rendered`, `animated`, `fallback`, `missing`, or `unsupported`.
- Stage compatibility reports include BG controller rows with bounded/unsupported status, timing, ctrl ids, resolved target layer labels, raw params, unsupported type notes, and fallback notes for exact parity gaps.
- Placeholder runtime background bands remain as a fallback when a stage BG sprite is missing or not decodable.

## Compatibility Profiles / IKEMEN Scanner

- Character compatibility reports include explicit profile metadata: `native-runtime`, `mugen-1.0`, `mugen-1.1`, and optional `ikemen-go-scan`.
- The first `ikemen-go-scan` implementation scans package paths and text files before runtime execution.
- Scanner-recognized IKEMEN-only features include `.zss` files, `.zss` state references, ZSS controller syntax, `.lua` files, `lua` / `luafile` / `luacode` hooks, Lua `hook.add` / `hook.run` / `hook.runFirst` calls, `config.json`, path-only screenpack `select/system/fight/storyboard/logo/intro/ending.def` files, select/story `unlock` expressions, `commandlist` references, `IkemenVersion`, selected IKEMEN-only controller names such as `AssertCommand`, `AssertInput`, `AssertAnalogVector`, `LifebarAction`, `ModifyPlayer`, `ModifyStageVar`, `ModifyBGCtrl`, `CameraCtrl`, `Dialogue`, `TagIn`, `TagOut`, `MapSet`, and `MapAdd`, selected `AssertSpecial` flags, selected source-mapped extended trigger identifiers, model-stage assets, and named 3D/Z stage params such as `topz`, `botz`, `ztopscale`, `depthtoscreen`, `zoffsetlink`, `startz`, `model`, and `fov`.
- IKEMEN scanner findings are added to `CompatibilityReport.profiles.ikemen`, the exported compatibility JSON, the DebugPanel compatibility section, Studio Evidence, and the unsupported feature list as scanner-only facts.
- Scanner findings are labeled `Recognized + Unsupported`; no ZSS, Lua, rollback, netplay, model-stage runtime, or IKEMEN-only behavior is executed.

## CMD

- `[Command]` blocks with `name`, `command`, and `time`.
- `[Defaults]` `command.time`, partial `command.steptime`, `command.buffer.time`, and partial `command.buffer.hitpause`.
- Per-command `time`, partial `steptime`, `buffer.time`, and partial `buffer.hitpause` overrides.
- `[Remap]` button aliases are parsed before command tokenization; remapped commands show their original and resolved sequences in the UI, and blank remap targets disable the affected command path.
- Direction/button/modifier/separator/combo token classification.
- `CommandBuffer` stores inputs by frame, exposes a bounded debug history of recent samples, and can query active commands with duplicate-name alternatives.
- Hit-pause input samples can be buffered for commands that opt into `buffer.hitpause`; hit-pause samples are ignored for commands that opt out.
- Derived diagonals (`DF`, `DB`, `UF`, `UB`) from simultaneous cardinal input.
- Command matching collapses input into an 8-way direction so `DF` does not satisfy a plain `F` step unless the command uses `$F`.
- Repeated direction/button steps such as `F, F` require a fresh activation instead of matching a held input across adjacent frames.
- Plain direction/button inputs now require a fresh press/change within the buffer window instead of staying active for the full duration of a hold.
- Partial `/` hold matching for buttons and direction families, such as `/x` and `/$D`.
- Partial `~` release matching for buttons and directions, including common direction-release transitions such as `~D, DF`.
- Partial numeric charge matching on `/N` and `~N` modifiers, including command forms such as `~30$D, F, x`.
- Partial `steptime` matching rejects commands when consecutive matched steps are farther apart than the command's step window; when unset it falls back to the command's total `time`.
- `$` direction-family matching for commands such as `/$D`.
- Same-frame `+` button combos.
- Same-step `|` alternatives for directions/buttons, including alternatives inside a same-frame `+` combo such as `x+y|z`.

## CNS/ST

- `[Statedef N]`.
- `[State N, Name]`.
- `[State -1, Name]` state-entry controllers inside `.cmd` are parsed through the CNS parser.
- `triggerall` is preserved with index `0` so command/state-entry logic can separate always-conditions from numbered trigger groups.
- Initial compiler IR exists under `src/mugen/compiler/`: parsed CMD commands compile into `CommandIr`, trigger expressions compile into `ExpressionIr`, controllers compile into `ControllerIr`, and character state data can produce a `RuntimeProgramIr` plus `CompileReport`.
- `MugenCharacterLoader` caches one `RuntimeProgramIr` per loaded character. Imported runtime fighters carry that compiled program, and `PlayableMatchRuntime` uses cached `StateProgramIr` / `ControllerIr` for active-state controllers, State -1 setup/routing, and trigger evaluation. `StateProgramExecutor` now dispatches compiled controllers into change-state, change-anim, shared runtime-controller, side-effect, or unsupported families. `CompatibilityReport` derives compiled state/controller/trigger/routable counts from the same compiler report instead of maintaining a separate controller-support list. Imported runtime sessions also expose `executedOperations` so traces and exports can prove when side effects such as `HitDef`, `Target*`, `Pause`/`SuperPause`, `Projectile`, `Helper`, `Explod`, `HitFall*`, and `FallEnvShake` came through typed `ControllerOp` data. Larger visual/combat side-effect controllers still run from parsed CNS params via `controller.source` until they move into dedicated typed systems.
- Partial expression/runtime triggers include `time`, `stateno`, `statetime`, `animtime`, `anim`, `life`, `power`, `alive`, `animelem`, `ctrl`, `statetype`, `movetype`, `physics`, `random`, `vel x/y`, `pos x/y`, `p2bodydist x/y`, `p2dist x/y`, `p2statetype`, `p2movetype`, `var(n)`, `fvar(n)`, `sysvar(n)`, `animelemtime(n)`, `selfanimexist(n)`, `const(...)`, `const720p(...)`, `gethitvar(...)`, `ifelse(...)`, `abs(...)`, `hitdefattr(...)`, `movecontact`, `movehit`, `hitshakeover`, `hitover`, bounded runtime-backed `inguarddist`, and `command = "name"` / `command != "name"` inside simple composite expressions.
- `AnimElemTime(n)` is runtime-backed in `PlayableMatchRuntime`: it uses the current AIR action, current frame index, current frame elapsed value, and frame durations to return time relative to element `n`. Exact MUGEN/Ikemen loop, invalid-element bottom value, and tick-order parity are still incomplete.
- `HitShakeOver` and `HitOver` are runtime-backed in the bounded two-actor path: `HitShakeOver` becomes true after hit pause ends, and `HitOver` becomes true after hit stun/guard stun ends. `GetHitVar(yaccel)` has a bounded default for Common1 airborne get-hit gravity, and `GetHitVar(slidetime)` / `GetHitVar(ctrltime)` are runtime-backed from parsed `guard.slidetime` / `guard.ctrltime` on `HitDef` and `Projectile` guard results when present, with bounded `0` fallbacks when absent. This is enough for the current Common1 stand get-hit, fall-branch, stand guard-hit `150 -> 151`, and crouch guard-hit `152 -> 153` gates, but it is not exact IKEMEN/MUGEN get-hit or guard timing parity.
- `InGuardDist` is runtime-backed in a bounded two-actor form: the opponent must have a current attacking move in a guard-distance window, the defender state must be eligible for the move's `guardflag`, the attack box is expanded by parsed `guard.dist` or a bounded fallback, and the expanded box must overlap the defender's current hurtbox. This supports explicit guard-start trigger evidence plus first bounded automatic guard-start and guard-end bridges for imported defenders holding back when defender-owned Common1-style states such as `120`/`130`/`140` exist. It does not claim exact proximity guard, exact guard-end timing, guard effects, or air guard parity.
- Trigger expressions normalize common MUGEN range syntax such as `GetHitVar(animtype) = [3,5]` into explicit comparisons for partial runtime/report support.
- Controller type, triggers, and params.
- Basic typed fields: `type`, `movetype`, `physics`, `anim`, `ctrl`, `velset`.
- Partial numeric controller params can evaluate simple expressions, including arithmetic, comparisons, `Const(...)`, and state-type checks.
- `PauseSystem` now owns the current partial `Pause`/`SuperPause` semantics outside `PlayableMatchRuntime` and can consume typed `pause:*` `ControllerOp` data: bounded `time`, source `movetime`, `darken` telemetry/rendering, `poweradd` deltas, pause snapshots, deterministic per-frame countdown, and runtime trace operation evidence. `RuntimeTraceGate` can require aggregated `matchPause` snapshot evidence plus actor/effect advance and freeze evidence, so the synthetic SuperPause traces now prove the typed operation produced a visible match-freeze snapshot with source actor/state, darken, remaining frames, source `movetime`, opponent freeze, and a bounded projectile effect route that advances during source `movetime` before freezing afterward.
- `AudioEventSystem` now owns the current partial `PlaySnd`/`StopSnd` event creation outside `PlayableMatchRuntime`; the Web Audio adapter consumes those events from actor snapshots.
- `EnvShakeSystem` now owns the current partial `EnvShake`/`FallEnvShake` event creation and deterministic camera-shake snapshot math outside `PlayableMatchRuntime`; compiled `FallEnvShake` controllers emit typed operation evidence when a fall shake event is produced, and Three.js consumes the resolved camera offset from the stage snapshot.
- `SpriteEffectSystem` now owns the current partial `SprPriority`, `PalFX`, `AfterImage`, and `AfterImageTime` parsing/ticking outside `PlayableMatchRuntime`; Three.js consumes the resolved actor sprite priority, material tint/opacity, and ghost-trail samples from actor snapshots.
- `EffectActorSystem` now owns the mutable per-fighter store for current partial `Explod`, `Helper`, and `Projectile` actors: serial allocation, bounded spawn lists, advance/removal mutations, hit-removal pruning, snapshot handoff, and read-only store summaries. `RuntimeEffectActorWorld` wraps those stores behind a world-style contract with separate active-effect and presentation-effect advance passes plus a bounded projectile-combat handoff, and `MatchWorld` creates/injects that contract into `PlayableMatchRuntime`; the integrated runtime still resolves owner/action/position data, but no longer owns three independent effect arrays or stores raw effect actor stores on `FighterMatchState`. `MatchWorld` exposes those summaries as `effectStores` in the actor registry so Debug Studio, QA bridge, and trace artifacts can compare the producer store with rendered effect snapshots.
- `ExplodSystem` now owns the current partial non-colliding `Explod` effect actor lifecycle outside `PlayableMatchRuntime`: typed `explod` `ControllerOp` data when compiled, fallback controller-derived id/removetime/facing/sprpriority/trans, animation frame advance, typed `RemoveExplod` operation data, actor identity metadata, and effect actor snapshots for Three.js.
- `HelperSystem` now owns the current partial visual `Helper` actor lifecycle outside `PlayableMatchRuntime`: typed `helper` `ControllerOp` data when compiled, fallback controller-derived id/name/removetime/facing/sprpriority, animation frame advance, stage-bounded removal, owner sprite metadata, actor identity metadata, and effect actor snapshots for Three.js.
- `ProjectileSystem` now owns the current partial colliding `Projectile` actor lifecycle outside `PlayableMatchRuntime`: typed `projectile` `ControllerOp` data when compiled, fallback controller-derived projid/removetime/facing/sprpriority/trans/velocity/damage/hit timing, guard damage/flag/pause/stun/slide/control/velocity/distance metadata, bounded `projpriority`, animation frame advance, stage-bounded removal, hitbox projection, owner sprite metadata, actor identity metadata, and effect actor snapshots for Three.js. `ProjectileCombatSystem` now owns the bounded projectile contact/reject/HitOverride/hit-or-guard/cleanup loop plus bounded projectile-vs-projectile trade/cancel through `RuntimeEffectActorWorld`, reusing `CombatResolver` for attack/defense scaling and held-back guard results; exact priority classes, cancel/remove animations, guard effects, helper-owned projectile rules, and MUGEN/IKEMEN timing parity remain unsupported.
- `TargetSystem` now owns the current partial target-memory, target-binding lifecycle, and simplified Target* controller side effects outside the main `PlayableMatchRuntime` controller loop: bounded recent target records, MUGEN-style wildcard id matching, target expiry, `TargetDrop` filtering, target binding durations, bound-position resolution, and partial `TargetLifeAdd`/`TargetPowerAdd`/`TargetVel*`/`TargetFacing`/`TargetBind`/`TargetState` application through a small runtime-system contract. `TargetState` still delegates state-entry validation and custom-state ownership back to the match runtime.
- `CombatResolver` now owns the current partial renderer-independent combat decision helpers outside `PlayableMatchRuntime`: facing-aware box projection, collision overlap checks, `HitBy`/`NotHitBy` eligibility, `HitOverride` lookup, simplified guard detection, attack/defense damage scaling, and hit/guard result math used by both direct attacks and bounded projectile combat. State transitions, target memory side effects, logging, and exact MUGEN/IKEMEN hit ordering still remain in bounded runtime-system contracts rather than full VM parity.

## Sprite Atlas Pipeline

- `AtlasSpriteProvider` can consume `sprite-atlas-builder` output: `sprite-sheet-alpha.png` plus `manifest.json.frame_layout`.
- `SffSpriteProvider` can consume decoded SFF v1 and SFF v2 archives from the browser loader.
- `CompositeSpriteProvider` routes selected group ranges to atlas-backed characters, supports owner-specific lookups for imported custom-state rendering, and falls back to procedural sprites for unsupported groups.
- Loaded decoded SFF routes take priority for their sprite groups so imported runtime actors can render their own art.
- Default action mapping is prepared for fighting rows: `idle`, `crouch`, `walk-forward`, `jump`, `punch`, `kick`, `hitstun`.
- The imagegen-sheet normalizer uses one character-level reference scale, so crouch/jump frames are not enlarged by independent per-frame fitting.
- Individual rows can be regenerated as new imagegen art and inserted with `scripts/replace_atlas_state_from_imagegen_row.py`; this is the preferred fix for bad locomotion poses over cutting or mechanically repairing old frames.
- The inspector fallback still uses procedural sprites when imported SFF images cannot be decoded.

- Full state-controller execution beyond the hand-authored playable subset and imported CMD State -1 bridge.
